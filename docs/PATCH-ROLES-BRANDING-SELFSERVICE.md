# Patch — Roles, Self-Service Passwords, Account Provisioning, Rebrand, PDF Logo

## What this patch covers

1. **Role-based dashboards** — `/` now renders differently depending on the
   logged-in user's role group (admin / governance / committee / staff /
   self-service), instead of one generic tile grid for everyone.
2. **Self-service password reset** — logged-in users can change their own
   password; logged-out users can request a reset link by email and set a
   new one.
3. **Account auto-provisioning for Members and Staff** — a "Create Login
   Account" action on the Member and HR Employment Contract screens, via
   one shared endpoint that doesn't modify either bundle's existing
   controllers.
4. **Visual rebrand** — green sidebar, brand-green link colors app-wide
   (including native `<a>` tags, not just Mantine's `<Anchor>`).
5. **Logo on every generated PDF report.**

Every change was scoped as tightly as the request allowed — see the
"Blast radius" note under each section.

---

## 1. Role-based dashboards

**Blast radius: `pages/DashboardPage.tsx` and a new `pages/dashboards/`
folder only.** No routes changed, no other screens touched.

Five variants, resolved from the user's roles (`resolveDashboardRoleGroup`
in `pages/dashboards/roleGroups.ts`), in priority order:

| Group | Roles | What's different |
|---|---|---|
| `admin` | `super_admin` | Extra stats row (user count, module/screen/role totals) above the module grid |
| `governance` | president, VP, secretary_general, treasurer, director_technical_affairs, board_director, ceo | Standard module grid, governance-framed copy |
| `committee` | commission_president, hr_committee, electoral_commission, legal_commission, conflict_resolution_organ, audit_organ | Standard module grid, committee-framed copy |
| `staff` | hq_business, hq_sport | Standard module grid, HQ-framed copy |
| `selfService` | everyone else (member, honorary_member, fan_club_rep, player, partner, guest) | Simplified single-column list instead of a dense grid, friendlier copy |

A user with multiple roles gets the richest applicable dashboard (e.g. a
`hr_committee` member who's also a plain `member` sees the committee view).

This is presentation logic, not route redirection — everyone still lands
on `/`, and `navConfig.ts`'s existing role-filtering (unchanged) still
decides which modules actually show up in the grid/list and the sidebar.

## 2. Self-service password reset

**Blast radius:** new `PasswordController`, three new routes, one new
frontend modal (`ChangePasswordModal`, opened from the existing user
menu), two new public pages (`ForgotPasswordPage`, `ResetPasswordPage`),
one link added to `LoginPage`. `AuthController` untouched.

- **Logged in:** user menu → *Change Password* → current + new password,
  `PUT /api/auth/password`.
- **Logged out:** *Forgot password?* on the login screen →
  `POST /api/auth/forgot-password` (always returns the same generic
  message, whether or not the email exists — doesn't leak account
  existence) → emails a reset link → `/reset-password?token=...&email=...`
  → `POST /api/auth/reset-password`.

Uses Laravel's built-in password-broker (`Password::sendResetLink` /
`Password::reset`), so it relies on the **`password_reset_tokens` table
that ships in Laravel 11's default `users` migration** — if you generated
your `users` table any other way, confirm that table exists.

The reset email's link is generated by a new
`ResetPasswordNotification` (not Laravel's default one), because the
default assumes a named Blade route (`password.reset`) that doesn't exist
in an API-only app — this one points straight at the SPA's
`/reset-password` route via `config('app.frontend_url')`, same pattern
already used by `WelcomeRegistrationNotification`.

## 3. Account auto-provisioning (Members & Staff)

**Blast radius:** one new migration (adds `user_id` to
`hr_employment_contracts` — `members` already had it since Bundle 1), one
new standalone controller (`AccountProvisioningController`), one new
frontend modal (`ProvisionAccountModal`), and a single additive column +
modal-open button on `MemberApplicationsPage` and
`HrEmploymentContractsPage`. **No existing controller logic was changed** —
`MemberController` and `HrEmploymentContractController` are untouched.

`POST /api/account-provisioning/{sourceType}/{sourceId}` (sourceType is
`member` or `hr-employment-contract`) either creates a new account with a
random temporary password, or — if the email already belongs to an
existing user — links that existing account to the record instead
(no new password generated, no duplicate accounts). Either way it assigns
the roles you pick and links back via `user_id`.

**Committees/commissions are not included** — Bundle 8 modeled commission
*pillars* and their work plans/KPIs, not individual commission membership,
so there's no source table to provision from yet. `GET
/api/account-provisioning/sources` returns the list of what's actually
available (`["member", "hr-employment-contract"]` today) so the frontend
never has to guess or silently fail. Adding a third source type later is
one entry in `AccountProvisioningController::SOURCES` — no other code
changes needed.

## 4. Visual rebrand

**Blast radius:** `theme/theme.ts` (Anchor defaults added), new
`styles/global.css` (imported once in `main.tsx`), `AppLayout.tsx`
(navbar background + NavLink styling). No other component files touched.

- Sidebar background: `#004d00` (brand green, darkened for contrast with
  white text).
- Active nav item: white background, dark green text — the clearest
  affordance for "you are here" against a dark sidebar.
- **All links, everywhere** — native `<a>` tags *and* Mantine `<Anchor>` —
  now default to brand green (`#006400`), a darker green on hover
  (`#004d00`), and a distinguishable muted green for `:visited` links
  (`#3d7a3d`) instead of browser-default blue/purple. This is a genuine
  global CSS change (`styles/global.css`), so it applies inside every
  screen's content area too, not just the sidebar.

## 5. Logo on PDF reports

**Blast radius:** `layouts/report.blade.php` (the one shared header every
report extends) and one new helper class. No per-module report view
needed changes.

`App\Support\BrandAssets::logoDataUri()` reads `public/images/kiyovu-crest.png`
and embeds it as a base64 data URI — DomPDF can't reliably fetch external
URLs, so this avoids that whole class of "logo doesn't show up in the PDF"
problem. **The crest file needs to exist at that exact path** — it's
included in this patch's zip; if you're applying this by hand, copy your
crest to `public/images/kiyovu-crest.png`. If the file is missing, the
helper returns `null` and the report still generates — just without the
logo — rather than crashing report generation.

## What's still not done, stated plainly

- **Commission/committee membership still isn't a modeled table**, so
  "committee members get an account on registration" only works for
  Members and Staff right now, not commission appointees — flagged in
  §3 above, not silently skipped.
- **No email verification step** on newly provisioned accounts — they're
  created `active` immediately. If that's a concern, `status_id` could
  default to `pending` instead and require an admin activation step; not
  implemented here since it wasn't requested.
- **The `/profile` link in the user menu still points nowhere** — it was
  already broken before this patch (Phase 2 never built a Profile page)
  and is out of scope here; "Change Password" now covers the immediate
  need without a full profile screen.
